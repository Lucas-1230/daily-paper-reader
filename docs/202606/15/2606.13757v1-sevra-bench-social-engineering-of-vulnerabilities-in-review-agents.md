## 摘要
大型语言模型（LLM）审查者正越来越多地用于拉取请求（PR）工作流程中，其批准意见有助于决定哪些代码合并到仓库中。这引发了一个静态漏洞检测或代码生成基准测试无法解决的问题：当攻击者同时控制代码更改和附带的PR文本时，自动化审查者能否拒绝恶意贡献？我们引入了SEVRA-BENCH（审查代理中漏洞的社会工程学），这是一个基准测试，用于衡量自动化审查者批准此类对抗性拉取请求的频率。SEVRA-BENCH中的每个恶意PR都基于一个真实的项目提交，该提交先前修复了通用漏洞与暴露（CVE）数据库中列出的一个漏洞。我们自动逆向该修复以恢复原始漏洞代码，并将其包装在15种社会工程学框架之一中作为拉取请求提交，这些框架在所做的声明、提供的支持证据、传达的紧迫性、先前批准的信号以及对权威的诉求等方面有所不同。SEVRA-BENCH包含1,062个恶意PR，这些PR来自2025年CWE前25名中前10项条目所关联的CVE修复。在真实场景下，我们评估了8个当前LLM作为代码审查代理，处理引入先前在公开披露中报告的漏洞的PR。我们的结果揭示了闭源和开源模型在安全能力上的显著差距。我们希望SEVRA-BENCH能成为推进开源模型并缩小这一差距的宝贵资源。

## Abstract
Large language model (LLM) reviewers are increasingly used in pull-request (PR) workflows, where their approvals help decide which code is merged into a repository. This raises a question that benchmarks for static vulnerability detection or code generation do not address: can an automated reviewer reject a malicious contribution when the attacker controls both the code change and the accompanying PR text? We introduce SEVRA-BENCH (Social Engineering of Vulnerabilities in Review Agents), a benchmark that measures how often an automated reviewer approves such adversarial pull requests. Each malicious PR in SEVRA-BENCH is built from a real project commit that previously fixed a vulnerability listed in the Common Vulnerabilities and Exposures (CVE) database. We automatically invert that fix to restore the original vulnerable code and submit it as a pull request wrapped in one of 15 social-engineering framings, which vary the claims made, the supporting evidence, the urgency conveyed, signals of prior approval, and appeals to authority. SEVRA-BENCH contains 1,062 malicious PRs drawn from Common Vulnerabilities and Exposures (CVE)-linked fixes across the top 10 entries of the 2025 Common Weakness Enumeration (CWE) Top 25. In a realistic setting, we evaluate 8 current LLMs as code review agents on PRs that introduce vulnerabilities previously reported in public disclosures. Our results reveal a sharp gap in security capabilities between closed- and open-source models. We hope SEVRA-BENCH will serve as a valuable resource for advancing open-source models and narrowing this gap.

### 2. 方法论
- **核心思想**：将真实的CVE漏洞修复**逆向操作**以恢复原始漏洞代码，并为其配上不同风格的社会工程学PR描述，构建对抗性测试样本；然后让LLM代理通过真实的API与Git仓库交互，做出“批准合并”或“拒绝”的决策。
- **关键技术与流程**：

### 6. 主要结论与发现
- **闭源模型显著领先**：Claude Opus 4.7总体拒绝率高达97.6%，GPT-5.5为95.2%，在几乎所有CWE和框架下均能稳健识别恶意PR。两者对SQL注入、越界写入等关键弱点实现100%拒绝。
- **开源模型严重落后**：DeepSeek V4-Flash拒绝率53.4%，Kimi K2.5为52.3%，GPT-5.4-nano仅35.6%。开源与闭源模型间的平均拒绝率差距高达45个百分点。
- **框架影响在弱模型中更显著**：同一开源模型在不同社会工程策略下拒绝率波动极大（例如DeepSeek对“诉诸权威”100%拒绝，但对“预先批准信号”仅18.2%拒绝）。闭源模型则几乎不受叙事框架影响。
- **安全理由率揭示“偶然拒绝”**：较弱模型即使拒绝PR，其拒绝理由往往是安全性之外的考量（如代码风格），安全理由率远低于闭源模型。尤其在Use-After-Free（CWE-416）等疑难漏洞上，SRR极低。

### 7. 优点
- **贴近真实攻击**：使用经验证的CVE修复逆向生成漏洞代码，而非模型生成，保证漏洞的真实性与隐蔽性。
- **社会工程维度系统化**：15种框架基于社会工程学文献和真实攻击案例精心设计，且与代码diff分离，可孤立测量叙事操控效果。
- **交互式、动态评测**：PR被部署在Git实例中，代理必须通过API探索仓库，模拟实际审查流程，而非静态代码分析。
- **全面且细粒度的分析**：不仅给出整体拒绝率，还深入揭示为何拒绝（安全理由率）、推理成本差异、框架脆弱性等，对模型选择和安全改进具有指导意义。

### 8. 不足与局限
- **数据污染风险**：闭源模型可能在训练中见过漏洞修复记录，导致记忆而非推理出漏洞，可能高估其真实安全能力。
- **饱和效应**：前沿闭源模型在SEVRA-BENCH上接近满分，未来可能难以区分更强的模型，削弱基准的区分度。
- **良性PR库单一**：仅包含明确的良性安全修复PR，未涵盖有争议的重构或混合权衡场景，可能低估误拒绝率（false decline）。
- **脱离丰富语境**：代理仅通过Git API交互，缺少作者声誉、项目历史、团队讨论等人类审查员可用的社会信号，评估可能过于理想化。
- **单轮攻击设定**：未模拟攻击者根据审查反馈迭代修改PR的多轮适应攻击，静态攻击无法反映更高级的交互式威胁。
- **算力与资源未说明**：缺少具体推理成本数据，难以评估部署可行性与环境差异。
