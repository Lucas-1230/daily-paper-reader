## 摘要
安全智能体的评估通常衡量在充足推理预算下的峰值攻击能力，侧重于漏洞发现、漏洞利用开发、渗透测试和CTF竞赛完成度。此类衡量标准虽有用但并不全面：在实际安全运营中，每个推理步骤、工具调用、遥测查询和信息丰富请求都会消耗预算。我们通过成本—成功视角，在进攻性Cybench挑战和防御性Splunk BOTS v1调查挑战上评估语言模型安全智能体。我们不再仅报告最佳情况下的成功率，而是在固定成本水平上比较不同模型，并将表现分解为推理支出和工具支出。结果显示，红队和蓝队任务呈现不同的规模扩展模式。进攻性CTF性能会随着额外的测试时计算而提升，且扩展后的开源权重模型能接近前沿的专有系统，同时保持成本竞争力。防御性SOC调查并不以相同方式扩展：成功更依赖于严格的工具使用、遥测导航和选择性信息丰富，而不仅仅取决于原始推理预算。我们认为，安全智能体基准测试应当衡量经济效率和运营适配性，以及任务成功率。成本感知、原生SOC的评估能更清晰地展示哪些模型在当前具有实际效用，以及防御智能体仍需在哪些方面进行改进。我们提供一个展示结果的交互式网站：https://evals.frontier.security。

## Abstract
Security-agent evaluations commonly measure peak offensive capability under generous inference budgets, emphasizing vulnerability discovery, exploit development, penetration testing, and CTF completion. Such measurements are useful but incomplete: in operational security, every reasoning step, tool call, telemetry query, and enrichment request consumes budget. We evaluate language-model security agents through this cost-success lens on offensive Cybench challenges and defensive Splunk BOTS v1 investigation challenges. Instead of reporting only best-case success, we compare models at fixed cost levels and decompose performance by inference spend and tool spend. Our results show distinct scalingregimes for red- and blue-team tasks. Offensive CTF performance improves with additional test-time compute, and scaled open-weight models can approach frontier proprietary systems while remaining cost-competitive. Defensive SOC investigation does not scale in the same way: success depends more heavily on disciplined tool use, telemetry navigation, and selective enrichment than on raw reasoning budget alone. We argue that security-agent benchmarks should measure economic efficiency and operational fit alongside task success. Cost-aware, SOC-native evaluations provide a clearer picture of which models are practically useful today and where defensive agents still need to improve. We present an interactive website with our results https://evals.frontier.security.

## 2. 论文提出的方法论

- **核心思想：成本-成功（Cost-Success）评估框架**
    - 将**成本**作为比较攻击和防御智能体的通用轴心。核心问题变为：在固定的经济预算下，一个模型能提供多少能力？
    - 区分两种成本：**推理成本**（模型Token消耗）和**工具成本**（如威胁情报API调用费用），并进行精细核算。

- **关键技术细节**
    - **统一评估框架**：使用 Inspect 评估框架构建 ReAct 风格的智能体，攻击和防御任务仅通过切换不同的工具集来实现，保证了评估的一致性。
    - **预算控制**：
        - **前瞻性控制**：在运行中设定总美元成本上限（如$0.80, $2.10），超过则任务失败。
        - **回顾性分析**：对已完成的运行轨迹，施加一个更低的事后成本上限来复盘，以观察性能随预算变化的“净空”（headroom）。这避免了混入不同运行轨迹的干扰。
    - **成本核算**：模型推理成本基于API定价；BOTS v1 防御任务中，`Brave Search`（$0.005/次）和 `WhoisXMLAPI`（$0.0258/次预览）等工具调用被计为额外成本。
    - **工具纪律**：区分工具调用的数量与质量，而非简单统计。

## 3. 实验设计

- **基准与数据集**
    1.  **攻击场景 (Cybench)**: 一个包含40个CTF挑战的基准，涵盖Web安全、逆向工程、密码学等领域。智能体需要通过执行 bash/python 命令找到并提交 flag 字符串。
    2.  **防御场景 (Splunk BOTS v1)**: 一个公开的SOC调查数据集，包含真实的Splunk日志。智能体需要像安全分析师一样，通过 SPL 查询、事件检查、外部情报查询来回答31个计分问题。
- **对比模型**
    - **前沿模型**: Claude Fable 5, Claude Opus 4.8, GPT-5.5, GPT-5.6 系列 (Sol, Luna, Terra)。
    - **开源/开放权重模型**: DeepSeek v4 Flash, DeepSeek v4 Pro。
    - **设置**: 对比了模型在不同推理努力（高努力 vs 默认）下的表现，并记录了因内容审核策略导致的“拒绝回答”情况。
- **去污染控制**
    - 鉴于 BOTS v1 是公开旧数据集，论文专门设置了“**无工具**” (no-tools) 对照组，即仅向模型提供问题文本，观察其是否能单凭训练记忆回答问题，以此作为基准污染的参照指标。

- **实验组数**：

- **充分性与公平性**：
    - **充分**：实验设计全面，将性能分解为“推理花费”和“工具花费”，并从固定成本、预算扩展净空、工具调用效率等多个角度剖析了攻击和防御任务的不同扩展特性，超越了简单的“排行榜”式对比。
    - **客观公平**：采用统一的评估工具（Inspect）和一致的智能体架构（ReAct）。成本核算方法公开透明，特别是回顾性分析方法，避免了简单比较不同运行结果的谬误。去污染实验体现了对结果公正性的审慎态度。

## 6. 论文的主要结论与发现

- **攻击与防御的扩展模式截然不同**：
    - **攻击性任务 (Cybench)**：成功率随推理预算和 Token 消耗的增加而明显提升。开源模型（如 DeepSeek v4 Flash）在成本远低于前沿模型的情况下，可达到有竞争力的成功率，证明了在攻击场景下“堆计算”是有效的。
    - **防御性任务 (BOTS v1)**：成功率**不**随预算线性提升。最佳性能（Claude Opus 4.8）是在较低成本和极少工具调用次数下达成的。高工具调用量往往意味着搜索效率低下，甚至会拉低分数。在这里，**工具使用纪律和信息导航能力**比原始推理预算更重要。
- **成本感知评估的必要性**：单纯的成功率排名会误导实践决策。经济效率（如 `$/1k BOTS points`）和运营成本是衡量模型现实可用性的关键指标。
- **基准污染的警示**：BOTS v1的“无工具”实验显示，部分模型能“背诵”大量答案，这严重干扰了对模型真实调查能力的评估，证明了在公开基准上运行去污染控制是强制性的。

## 7. 优点：方法或实验设计亮点

- **提出了一个新颖、实用的评估范式**：将“成本”作为第一性原理引入安全智能体的评估，紧密贴合了真实世界的运营决策。
- **工作负载分解**：清晰地区分了攻击性CTF与防御性SOC调查这两种截然不同的任务类型，并指出了它们对模型能力需求的根本差异，即“为什么”防御任务不能简单套用攻击任务的扩展规律。
- **精细且审慎的分析方法**：回顾性预算分析、成本分解（推理 vs. 工具）、工具效率分析等一系列方法，使得结论更具洞察力和可靠性。去污染控制的设置也提升了研究的严谨性。
- **建立了可复现的基准**：基于公共数据集和开源框架（Inspect），并提供了交互式网站和详细的评估日志，可复现性高。

## 8. 不足与局限

- **防御任务覆盖有限**：仅使用了 Splunk BOTS v1 一个公开的数据集，且其年代久远，存在数据污染问题。研究结论是否能推广到其他更现代、更复杂的SOC环境（如BOTS v2/v3）有待验证。
- **代理架构单一**：所有实验基于 ReAct 风格的智能体，未探索其他智能体架构（如Plan-Act）在成本-成功维度上的表现。
- **成本计量不完全**：防御成本中未计入运行 Splunk 基础设施、分析师复核时间等隐性成本。外部工具的成本也只计算了部分。
- **实验的观察性质**：实验为观测性而非前瞻性的随机对照试验。虽然回顾性分析模拟了预算限制，但与真正的预算受限运行仍有差异。部分模型（如 DeepSeek v4 Flash）因达到消息数量上限而被截断，可能低估了其高计算量下的表现。
- **拒绝回答干扰**：部分模型（尤其是 GPT-5.6 系列和 Claude Fable 5）因内容安全策略的高拒绝率，导致其攻击任务的部分性能评估更像是“安全策略测试”而非“能力测试”，数据解读需格外谨慎。
